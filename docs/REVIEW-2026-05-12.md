> 仓库：`github.com/6547709/goct`
> 范围：`git pull` 后最新 `main`（commit `a3f2bb5`，已发布 `v0.2.0`）

本报告由 Senior Developer 视角对项目做静态审查。`goct` 的目标是用 govc 风格的语法面向 **CloudTower / SmartX ELF**（不是 vSphere），并把 CloudTower 独有能力一并纳入。下面分四块阐述：

1. 总体观感
2. **真实 Bug**（线上必现或在某些路径上一定出错）
3. 设计与工程问题（不一定崩，但破坏架构/安全/可维护性）
4. 改进建议（按优先级排序）

## 1. 总体观感

**优点**

- 三层架构（`cmd → service → adapter`）落地得很清晰，`adapter` 是唯一引用 SDK 的层，符合 SPEC.md 的反腐设计。
- 资源覆盖面广：VM / 快照 / 主机 / 集群 / 存储 / 网络 / VLAN / 任务 / 告警 / 用户 / 标签 / 模板 / Folder / PlacementGroup / 内容库 / 应用 / 部署 / License / Alert Rule / NTP / 集群设置 / Metrics / USB / GPU 等都有了。
- 通用解析器 `service.Resolve[T]` 用泛型，写得不错；`name vs id` 用 UUID + cuid 两种正则并行识别，符合 CloudTower 真实 ID 形态。
- Session 缓存 / 401 自动重登 / Task watcher / 分级 trace（含 HTTP RoundTripper 拦截）/ debug 日志，基础设施齐全。
- 单元测试在 `service`、`adapter`、`config`、`session`、`task`、`debug`、`output`、`flags` 都有，跑得通。

**需要正视的红线**

- 仓库里直接 commit 了 **35 MB 的 `goct_test` 二进制**（Mach-O arm64），并且**漏掉 `.gitignore`**。这条线必须修。
- `cloudtower-go-sdk/`、`cloudtower-skills/`、`govmomi/`、`packer-plugin-cloudtower/` 四个 **gitlink 目录**（mode 160000）但没有 `.gitmodules`，等于"半拉的子模块"，clone 下来全是空目录。
- `cmd/` 命令越来越胖，有的把 60% 业务逻辑（解析 cluster 名、解析 VLAN 名、组合参数）摊在 cmd 层，违反三层架构。
- `docs/CLOUDTOWER_API_STATUS.md` 与代码已经**严重不同步**（content_library_image / application / usb_device 实际已实现，文档仍标 ❌）。

## 2. 真实 Bug

### 🔴 Bug 1 — `--trace` 与 `--insecure` 互斥（高频，必出）

`pkg/adapter/client.go: newTransport`

```go
if insecure {
    hc.Transport = &http.Transport{TLSClientConfig: &tls.Config{InsecureSkipVerify: true}}
}
if customTransport != nil {
    hc.Transport = customTransport   // ← 直接覆盖 insecure 设置
}
```

在 `cmd/root.go` 里：

```go
traceTransport = debug.NewTraceRoundTripper(nil, opts.TraceLevel, os.Stderr)
```

`nil` 作为 base，`TraceRoundTripper` 内部回退到 `http.DefaultTransport`（**会做 TLS 校验**）。结果是只要用户同时开了 `--insecure --trace`，自签名 CloudTower 一定 TLS 失败。这条用户场景非常常见（内网调试）。

**修复**：在 `cmd/root.go` 构造 trace 时，先按 insecure 决定 base transport；或者把"组合 transport"的责任收回到 `adapter.newTransport`，让 insecure + custom 二者都能叠加。

```go
base := http.DefaultTransport
if cfg.Insecure {
    base = &http.Transport{TLSClientConfig: &tls.Config{InsecureSkipVerify: true}}
}
traceTransport := debug.NewTraceRoundTripper(base, opts.TraceLevel, os.Stderr)
```

### 🔴 Bug 2 — `task.ErrFailed` 永远命不中 exit code 4

`pkg/task/watcher.go` 定义了**自己的** `ErrFailed`：

```go
var ErrFailed = errors.New("task failed")
```

而 `cmd/helpers.go: errorExit` 检查的是 `adapter.ErrTaskFailed`：

```go
case errors.Is(err, adapter.ErrTaskFailed):
    os.Exit(4)
```

两个 sentinel 不是同一个变量。`Watch` 失败时会包 `task.ErrFailed`，最终走到 default → exit 1，`SPEC` 里承诺的"task FAILED → 退出码 4"**始终不会触发**。

**修复**：`pkg/task` 直接 `import` 并复用 `adapter.ErrTaskFailed`，或在 `errorExit` 里同时识别两者。前者更干净。

---

### 🔴 Bug 3 — `vm.create` 默认创建一块 10 GB SCSI 盘 + 一张 VIRTIO 网卡，且不可关闭

`pkg/adapter/vm.go: CreateVM`（行 137–168）硬编码：

```go
defaultDiskSize := int64(10 * 1024 * 1024 * 1024)
VMDisks: ...mountNewCreateDisks: [{name:"disk0", size:10G, bus:SCSI}]
VMNics:  []*VMNicParams{{Type:VLAN, Model:VIRTIO}}  // 没有 ConnectVlanID
```

问题：
- 用户调 `vm.create --name foo --cluster c1 --vcpu 2 --memory 2048` 会**默默拿到一块 10 G 磁盘**和一张未接入任何 VLAN 的网卡，没有提示。
- VLAN 没有 `ConnectVlanID`，CloudTower 会拒绝（实际 API 一般要求 NIC 必须挂到一个 VLAN）。线上很可能直接 422。
- 如果用户希望从零干净创建（不挂磁盘或自己后置 `vm.disk.add`），**没有任何 flag 可以关掉这两个默认项**。

**修复**：把 disks/nics 改为 `--disk size=10g,bus=SCSI`、`--nic vlan=<id>,model=VIRTIO` 的可重复 flag，未传时不下发任何盘/网卡（或只下发空数组让 CloudTower 报错），与 govc `vm.create` 的语义对齐。

### 🔴 Bug 4 — `ConvertToVM` 写死 `Name=""`，CloudTower 必拒

`pkg/adapter/vm.go:1245`：

```go
params.SetRequestBody([]*models.ConvertVMTemplateToVMParams{
    {ConvertedFromTemplateID: pointy.String(templateID),
```

`Name` 是模板转 VM 之后的目标 VM 名，调用必失败。

**修复**：从 service 层把 `name` 透传下来；如果用户没填，至少回退到 `template.Name + "-vm"`。

### 🔴 Bug 5 — `AbortMigrateAcrossCluster` 完全忽略 `vmID`

`pkg/adapter/vm.go:1223`：

```go
func (c *defaultClient) AbortMigrateAcrossCluster(ctx context.Context, vmID string) (TaskRef, error) {
    ...
    SetRequestBody(&models.AbortMigrateVMAcrossClusterParams{
        Tasks: &models.TaskWhereInput{},   // ← 空过滤，会把所有迁移 task 都尝试 abort
    })
```

入参 `vmID` 没有被用上，请求体里的 `Tasks` 是空 `TaskWhereInput{}`，等价于"全部"。这意味着这个命令在多 VM 并行迁移的场景会**误中**别的 task。

**修复**：要么按 SDK 规范把 task 关联到 VM ID，要么先查这个 VM 当前 in-progress 的迁移 task 再传。从 SDK 实际接口看，应该传 `Tasks.Vm.ID = vmID`（具体字段以 SDK 模型为准）。

### 🔴 Bug 6 — `UpdateDisk` 只更新 ID，其他参数全部丢

`pkg/adapter/vm.go:1144`：

```go
func (c *defaultClient) UpdateDisk(...) (TaskRef, error) {
    data := &models.VMUpdateDiskParamsData{
        VMDiskID: pointy.String(diskID),   // ← 只设置了 ID
    }
    // spec 里的 SizeBytes / Bus / Boot / IopsMax / ... 全部没有写入 data
    ...SetRequestBody(&models.VMUpdateDiskParams{
        Where: &models.VMWhereInput{ID: pointy.String(diskID)},  // ← 还把 disk ID 当成 VM ID 用
        Data:  data,
    })
}
```

两处都错：
1. `Where` 是 `VMWhereInput`，应当填 VM ID，结果填了 disk ID。
2. `data` 里所有用户传入的字段都没被设置，等价于"调用了一次什么都不改的 API"。

**修复**：补全 `data` 各字段；`Where` 改成 VM ID；service 层在调用前先 Resolve 出 VM。

### 🔴 Bug 7 — `UpdateVM` 不能把 description 改成空字符串

`pkg/adapter/vm.go:344`：

```go
if spec.Description != "" {
    data.Description = pointy.String(spec.Description)
}
```

CLI 没办法清空 description。这是经典的 `*string` 模式没贯彻彻底的问题：spec 字段应该是 `*string`，nil 表示"不变"，`""` 表示"清空"。`spec.Name` 同理。

### 🔴 Bug 8 — `math/rand` 在 `Migrate` 里使用，没有 seed，行为可预测

`pkg/service/vm.go:104`：

```go
selected := validHosts[rand.Intn(len(validHosts))]
```

Go 1.20 之后 global rand 默认已经是 nondeterministic（OK），但更重要的是这条"随机选目标主机"的语义本身值得商榷——govc 的 `vm.migrate` 在不指定 host 时是让 vCenter DRS 选，不是客户端自己 random。

**修复**：当 `hostID == ""` 时，应直接调 CloudTower 的"自动选择目标主机"接口（如果 SDK 支持），或要求用户必须显式指定。客户端做 round-robin 容易把 VM 全集中到一台空闲主机。

### 🟠 Bug 9 — `ListVMs` 未利用 SDK 的服务端过滤，`Resolve` 名称匹配会拉全表

`pkg/adapter/vm.go: ListVMs` 只识别 `NameContains / ClusterID / InRecycleBin`。`Resolve` 在 `service` 层用 `NameContains: idOrName` 查询，再在客户端 `n == idOrName` 精确匹配。

问题：

- `NameContains` 是模糊匹配，名字带 `-` `_` `.` 的常见场景会拉到一堆候选。再加上 CloudTower 单集群上 VM 上千的并不少见，每次 `vm.info <name>` 都触发一次大列表。
- 应当先尝试用 SDK 的 `Name=<exact>` 精确字段（`models.VMWhereInput.Name`）+ fallback 到 `NameContains`。

### 🟠 Bug 10 — `CreateVM` / `CreateFromTemplate` 把 spec.HA 写死为 true

`pkg/adapter/vm.go:144`：

```go
Ha: pointy.Bool(true),
```

`VMCreateSpec` 没有 HA 字段，命令层也没暴露 `--ha`，但又强行打开。在不开 HA 的小集群（或只装了单机版 CloudTower 测试环境）会报错。

**修复**：spec 加 `HA *bool`，nil 时不下发；CLI 对应 `--ha=true|false`。

### 🟠 Bug 11 — `UpdateNic` 的 `Where` 缺失

`pkg/adapter/vm.go:1130`：

```go
params.SetRequestBody(&models.VMUpdateNicParams{
    Data: data,   // ← 没有 Where
})
```

`VMUpdateNicParams` 通常需要 `Where: VMWhereInput{ID: vmID}`，再用 `data.NicID` 指定具体 NIC。当前 SDK 可能容忍，但语义上多 NIC 同 ID 时（理论上不会，但跨集群的极端场景）行为不定。

### 🟠 Bug 12 — Login 假设 token TTL = 8h，与 CloudTower 真实策略无关

`pkg/adapter/client.go:156`：

```go
return SessionToken{Value: tok, ExpireAt: time.Now().Add(8 * time.Hour)}, nil
```

注释也承认是假设。后果：如果 CloudTower 实际 TTL 短于 8h，缓存命中后请求 401 → 客户端只能靠 `client.New` 的 fallback 重登，每次都要"先失败一次"。

**修复**：检查 SDK login 响应是否带 `expires_in` / `expire_at`，没有就把 TTL 调短到比如 30 分钟，或者在 `Save` 时只保存 token，不写 ExpireAt，靠 401 触发的 invalidate 重登（现在已经有 fallback 路径，缩短到 30 min 的好处是减少首次失败延迟）。

---

### 🟠 Bug 13 — `IsID` 把 cuid 长度卡得太死，会误判长名字

`pkg/service/resolver.go:21`：

```go
var cuidRe = regexp.MustCompile(`^[a-z]{2}[0-9a-z]{23,25}$`)
```

cuid v1 标准是 25 字符（开头 `c`），cuid2 是 24 字符。这个正则要求 25–27 字符且前两位都是小写字母——cuid v1 完全匹配（开头 `c` 后面随机字母），但 cuid2 长度只有 24，会被判成 name。同时如果用户给 VM 取名叫 `aaaaaaaaaaaaaaaaaaaaaaaaa`（25 个 a，这种边角名字），会被误判成 ID 直接走 `Get`，找不到就报"not found"，这其实是个名字。

**修复**：CloudTower 的 ID 实际形态是 `cl` 开头 + 数字字母 23 字符 = 25 总字符（看了实际响应）。把正则收紧到 `^cl[0-9a-z]{23}$` 更准。

---

### 🟠 Bug 14 — IP 字段从字符串 split 拆分，逗号包含场景下拆错

`pkg/adapter/vm.go:778`：

```go
out.IPs = strings.Split(*v.Ips, ",")
```

CloudTower 的 `ips` 字段在多网卡时通常用逗号分隔，但偶发情况下也见过空格、分号。更稳妥的做法是先按 `,` 拆，再 trim，再 filter 掉空字符串。trim 已经做了，过滤空串没做，结果会出现空元素。

## 3. 设计与工程问题

### 3.1 仓库卫生

- **必须**从 git history 删除 `goct_test`（35 MB 二进制不该入库），并把 `goct_test` 加进 `.gitignore`。同样 `*_test` 这类编译产物也建议加。`test_all.sh` / `test_vm_features.sh` 没问题，但建议挪到 `scripts/` 子目录。
- 四个 gitlink 目录（`cloudtower-go-sdk` / `cloudtower-skills` / `govmomi` / `packer-plugin-cloudtower`）**要么补 `.gitmodules`，要么删掉**。当前状态会让任何 fresh clone 的人迷惑。
- `docs/CLOUDTOWER_API_STATUS.md` 必须做一次"代码 vs 文档"的对账，把已实现的标 ✅；建议改成由生成脚本扫 `pkg/adapter/*.go` 自动生成，避免手改漂移。

### 3.2 三层架构破窗

CLI 层混入业务逻辑的例子：

- `cmd/vm/create.go` 直接构造 `VMCreateFromTemplateSpec`，把 `--memory` 的单位换算（`MiB → bytes`）也放在 cmd 层；service 层应该负责 spec validation + unit conversion。
- 多个 `cmd/<resource>/register.go` 里写了拼接 `Where` 条件的小逻辑，本应下沉到 adapter。

建议把 service 层的接口提炼出来（每个资源一个 interface），让 cmd 只调 service 接口，**禁止 cmd 直接 import adapter**（除了类型）。可以用 `go vet` 自定义 lint 检查 `cmd/...` 是否 import 了 `pkg/adapter` 的非类型符号。

### 3.3 错误处理

- `adapter` 层的错误链没有把"原始 SDK error"完整透传。SDK 报 422 / 404 时大量上下文丢失，后续 debug 难定位。建议保留原始 `apierror`，并在 `errors.Is` / `errors.As` 上做 typed mapping。
- `errorExit` 的退出码映射只做了三个 sentinel；CloudTower 的 SDK error 可以提取 HTTP status，可以多映射几档：401 → 2、403 → 5、404 → 3、409 → 6、5xx → 7。脚本场景用得着。

### 3.4 安全

- Session cache 文件 0600 OK，但 `cacheDir()` 在 `XDG_CACHE_HOME` 不存在且 home 也不可用时落到 `/tmp`，多用户机器上（虽然路径含 sha1 不可猜）仍属可改进项。建议在落到 /tmp 时不缓存，强制每次重登。
- `--password` 通过 cobra flag 暴露，在 `ps -ef` 里能看到。建议加 `--password-stdin` / `-` 模式，从 stdin 读，govc 也是这样做的。
- `GOCT_PASSWORD` 走环境变量没问题，但应该在 README 提示用户**不要 commit 配置文件 `~/.goct.yaml`**。

### 3.5 govc 对齐度

对照 `govc-command.md` 的命令矩阵，goct 还有不少 govc 高频命令缺失或语义不同：

| govc 命令 | goct 状态 | 说明 |
|---|---|---|
| `find -type m -name pattern` | ❌ | govc 的统一 inventory find，goct 里只能各资源 ls |
| `vm.ip <name>` | ✅（`vm ip`） | 名字略不同，建议加 `vm.ip` 别名 |
| `vm.console`（VNC URL） | ✅（`vm vnc`）| OK |
| `vm.markastemplate` / `vm.markasvm` | ✅（`vm.convert_to_vm`）| 缺反向 mark template |
| `datastore.upload/download/cp/mv/mkdir` | ❌ | CloudTower 文件 API 缺，应至少标注 |
| `events`、`logs` | ❌ | CloudTower 有 audit log API 可补 |
| `permissions.*` | ❌ | CloudTower RBAC API 可补 |
| `metric.sample/info/ls` | ⚠️ | 已实现一部分，但命名 `vm.cluster` `vm.host` 这种 govc 风格 metric 没对齐 |

govc 的"路径式 inventory"（`/DC/vm/...`）在 CloudTower 里没有概念，硬模仿不划算，但可以提供 `--cluster <c>` + `--folder <f>` 这种 flag 组合达到类似效果。

### 3.6 CloudTower 独有能力

下面这些是 CloudTower 强项、govc 没有的，目前要么没实现要么命令名没暴露好：

- **快照计划 / 备份计划 / 备份恢复**（`snapshot_plan`, `backup_plan`, `backup_restore`） — 只列了 `ls/rm`，缺 `create/update/run`。
- **跨集群迁移**（`vm.migrate_across`） — 已实现，但 `abort` 有 Bug 5，需要修。
- **一致性组 / 放置组 / 反亲和** — `vm_placement_group` 已有但只 `ls/create/rm`，缺 `change`。
- **资源池 / 文件夹 / 标签 / Entity Filter** — Folder/Label OK，Entity Filter 全缺。
- **License 管理** — 已实现 `ls`/`update_deploy`，但缺 `apply`/`remove`/`activate`。
- **应用 / 部署 / 内容库镜像分发** — 已实现，文档没更新。
- **报警规则 + 通知策略**（`alert_rule`, `notification_policy`） — 只有 `alert_rule.ls`。
- **超融合存储池 / vSAN 配置 / iSCSI / NFS / NVMe-oF** — 全部缺。这是 SmartX ELF 的核心卖点，CLI 维度缺会很难看。
- **机架/Brick/Node 拓扑、可用区、数据中心** — 全部缺。
- **业务主机（vSphere ESXi 接管）相关 API** — CloudTower 接管异构虚拟化的卖点，全部缺。

### 3.7 测试覆盖

- `pkg/service/vm_test.go` 537 行已经不错，但是 `pkg/adapter/vm.go` 1265 行只有 `client_test.go` 是登录相关，**adapter 层 VM/Disk/NIC/CD-ROM 的具体 API 调用没有任何 mock 测试**。建议至少给"组装请求体"的部分写表驱动测试（不打真实 SDK）。
- `cmd/` 一个测试都没有，至少 `vm.create` / `vm.power` 这种关键命令要写 cobra 的 `RunE` 测试，验证 flag → spec 的转换。

### 3.8 输出层

- `pkg/output/columns.go` 181 行在新增很多列，但每个资源都"自己列一遍想要的列"，导致一致性差。可以抽一个 `OutputDescriptor` 概念，每个资源声明 `(列名, getter func(T) string)`，自动生成 table/json/csv。
- 缺 `--csv` 输出。脚本场景 `awk` 处理 table 比 CSV 麻烦。
